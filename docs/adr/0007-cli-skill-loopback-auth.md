# CLI skill auth via loopback callback; non-expiring CLI token

The `pagekit` CLI skill (a non-interactive one-shot script invoked by Claude Code) must authenticate against a deployed Pagekit instance. It previously stored the admin username/password in `config.json` and re-sent them on each login. We are removing stored credentials from the skill: on first operation the skill opens the browser to Pagekit's web login, and after login a JWT is passed back to the skill and cached locally for subsequent operations. The cached token must not expire.

## Decision

- **Token transport: loopback callback.** On first operation (or explicit `pagekit login`), the skill starts a local HTTP server on `127.0.0.1:<random port>`, opens the browser to `${host}/login?redirect_uri=http://127.0.0.1:PORT/cb&state=<random>`, and waits. After the user logs in, the web client redirects the browser to the loopback URL with `?token=<cli token>&state=<random>`; the skill validates `state` and caches the token.
- **Client-driven redirect.** `Login.tsx` reads `redirect_uri`+`state` from the query string; after a successful web login (or immediately if already authenticated) it calls a new `POST /api/auth/cli-token` endpoint and then `window.location`-redirects to the loopback URL. The server's existing `/api/auth/login` is unchanged.
- **`redirect_uri` allowlist.** Only `http://127.0.0.1` / `http://localhost` / `http://[::1]` (any port) are accepted; anything else is rejected to prevent token exfiltration via a crafted `redirect_uri`.
- **`state` binding.** Random per-flow `state` generated by the skill, validated on callback, prevents forged-callback token injection.
- **CLI-only non-expiry.** `/api/auth/login` keeps `expiresIn: '7d'` for the web session. The new `POST /api/auth/cli-token` (behind `authMiddleware`) mints a separate token with **no `expiresIn`** and a `type: 'cli'` claim. `authMiddleware` accepts either token type.
- **Revocation: rotate secret.** No server-side blocklist. `pagekit logout` clears only the local cache. To truly invalidate a leaked CLI token, rotate `config.jwtSecret` (which also invalidates all web sessions).
- **401 handling.** A 401 on a cached CLI token clears the cache and re-triggers the browser flow once; a second 401 is a fatal AUTH error.
- **Config shape.** `config.json` holds only `host`; env `PAGEKIT_HOST` overrides. `PAGEKIT_TOKEN` may inject a token directly (headless/CI escape hatch). `PAGEKIT_USERNAME`/`PAGEKIT_PASSWORD` are removed.
- **Headless.** If `open`/`xdg-open`/`start` is unavailable, the skill prints the full login URL (with SSH port-forward guidance) and keeps listening; `PAGEKIT_TOKEN` fully bypasses the browser.

## Rationale

- The skill is a non-interactive CLI, so it cannot prompt for a password. Loopback callback (RFC 8252) is the standard way to move a token from a browser to a local process without storing a long-lived secret in config.
- A separate non-expiring CLI token keeps the web session's 7d expiry intact (limits browser-theft exposure) while sparing the skill from re-triggering the browser every week — which matters because the skill may run unattended in a Claude Code session.
- Revocation via secret-rotation is acceptable for a single-admin, self-hosted tool (see ADR-0003); a jti blocklist's per-request DB check is not justified here.

## Considered Options

- **Token transport:** loopback callback (chosen) vs. manual copy-paste of token vs. device-code flow. Manual paste is two-step and clumsy for a one-shot skill; device code needs a polling endpoint and a code table.
- **Redirect owner:** client-driven (chosen) vs. server-driven 302. Server-driven would force `/api/auth/login` to serve both JSON and form/302, mixing concerns.
- **No-expiry scope:** CLI-only (chosen) vs. global. Global would make stolen browser `localStorage` a permanent admin credential.
- **Revocation:** rotate-secret (chosen) vs. jti blocklist vs. issued-token allowlist. Blocklist/allowlist add per-request persistence for a single-admin tool.
- **401:** auto-relogin once (chosen) vs. fatal "run `pagekit login`". Auto-relogin matches the existing UX and the "first operation triggers login" requirement.

## Consequences

- `authMiddleware` must tolerate two token shapes (`type: 'cli'` and the legacy web token); future revocation scoping can key off `type`.
- A leaked CLI token is valid until `jwtSecret` is rotated; this is documented as the revocation procedure.
- The skill's browser flow needs a timeout (5 min) and a headless fallback to avoid hanging unattended sessions.
